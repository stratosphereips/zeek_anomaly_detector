Zeek Anomaly Detector Explained

This page documents the full design of the tool: what data it reads, how it builds features, how it correlates Zeek logs through uid and fuid, which anomaly-detection techniques it applies to each log type, how the directory score is constructed, how the normal baseline works, and why those design choices were made.

Inputs: Zeek TSV and Zeek JSON
Modes: single log or whole directory
Core idea: one detector does not fit every Zeek log
Outputs: anomalies, directory score, JSON, PDF, summary line

Purpose

The tool is built to analyze Zeek logs as an investigation set, not just as isolated files. Its goal is to rank suspicious flows, transactions, files, events, and operational states, then summarize the whole directory as a directory-level maliciousness score.

What it is not

It is not a single global unsupervised model applied blindly to every Zeek schema. Zeek logs describe very different things: flows, web transactions, files, inventories, alerts, telemetry, and runtime state.

What it is

It is a multi-stage analysis pipeline with log-specific feature builders, log-specific scorers, cross-log context, and a final directory-level aggregation step.

Input Data

The detector accepts either one Zeek log file or a whole directory of Zeek logs. It auto-detects the on-disk format.

Supported file formats

  • Classic Zeek TSV logs with #fields headers
  • Line-delimited Zeek JSON logs

Common log types

  • conn.log, http.log, dns.log, files.log
  • ssh.log, tls.log, ssl.log
  • weird.log, notice.log, stats.log
  • known_hosts.log, known_services.log, software.log
Ignored by design: loaded_scripts.log is skipped completely. It reflects Zeek runtime configuration rather than network behavior, and it adds noise to summaries and plots without helping malicious-traffic detection.

Processing Pipeline

The tool follows a structured sequence so that log-specific scoring can still benefit from directory-wide context.

1

Discover logs

In directory mode, the tool enumerates .log files, skips ignored logs, and skips empty files cleanly rather than aborting the whole run.

2

Load data

Each file is loaded as a pandas DataFrame. TSV logs use Zeek headers when available. JSON logs are parsed as line-delimited records.

3

Build shared context

Before scoring any file, the tool constructs directory-wide context from all loaded logs. This creates per-uid aggregates and file metadata lookups used later by conn, http, files, dns, ssh, and other feature builders.

4

Engineer features per log type

Each log type gets a different feature builder. The builders generate numeric features, categorical rarity scores, cross-log counts, ratios, lexical measures, or time-series derivatives depending on what the schema means.

5

Apply a log-specific scorer

The tool picks a scorer per log family: multivariate Isolation Forest, hybrid DNS logic, hybrid connection scan logic, rarity scoring, time-series deviation scoring, or a fallback distance method when the feature space is too small.

6

Produce per-file anomaly rankings

Each file returns row-level scores, predicted anomalous rows, score percentiles, and metadata for printing, plotting, and JSON export.

7

Build a directory-level maliciousness score

The final score combines the strongest per-file anomaly concentration, cross-log overlap, weird/notice bonuses, HTTP-file overlap, and an explicit connection-behavior scan profile.

Cross-Log Correlation

A large part of the tool’s value comes from correlating related records across Zeek logs.

uid correlation

uid is Zeek’s transaction identifier. The tool uses it to aggregate context such as related connection bytes, HTTP counts, file counts, weird-event counts, SSH authentication attempts, duration, and connection-state rarity.

uid_log_types
uid_http_count
uid_files_count
uid_weird_count
uid_conn_bytes
uid_conn_duration

fuid correlation

fuid is Zeek’s file identifier. The detector uses it mainly to enrich http.log with linked file counts, linked file bytes, and linked MIME rarity from files.log.

linked_file_count
linked_file_bytes
linked_file_mime_rarity
Why this matters: a weak anomaly in one log can become much more meaningful when it shares a uid with odd HTTP behavior, a suspicious file transfer, or a weird event. This is one reason directory mode is more informative than analyzing a single file.

Feature Engineering

Features are engineered to match the semantics of each Zeek log. Some logs are true flows. Some are sparse metadata. Some are telemetry. Some are already alert-like.

Log Main engineered features Why those features matter
conn.log bytes, packets, duration, destination port, state rarity, history flags, short-connection flags, zero-payload flags, source-level port sweep behavior Captures scans, failed probes, bursty connection patterns, and odd multivariate flow combinations
http.log URI length, query depth, status rarity, request and response sizes, method rarity, linked file counts and bytes via fuid, related uid context Captures unusual web transactions, downloads, and HTTP-file relationships
dns.log query length, entropy, digit ratio, vowel ratio, unique-character ratio, DGA-like flags, repeated DGA-like source behavior, response absence, rejection indicators Separates suspicious lexical DNS behavior from common local-discovery noise
files.log total bytes, seen bytes, duration, MIME rarity, source flags, related HTTP and connection context Captures odd transferred files, unusual MIME types, and download context
ssh.log banner rarity, auth attempts, related connection context, weird-event context Captures sparse but useful metadata for SSH probing and odd negotiation behavior
tls.log / ssl.log cipher and version metadata, SNI-related rarity, basic session metadata, connection context Captures unusual TLS sessions and strange service combinations
weird.log / notice.log event-name rarity, notice linkage, related transaction context These logs already describe exceptional states, so rarity and prioritization matter most
stats.log bytes per packet, events per packet, queue pressure, TCP/UDP/ICMP shares, file and DNS pressure, reassembly pressure, memory and queue deltas Turns raw telemetry into operational pressure features instead of naive raw-counter outliers
inventory logs host, service, software, ARP, packet-filter rarity and drift-style indicators These logs are more useful for novelty and drift than for direct flow outliers

Models and Scoring Techniques

Different Zeek logs are scored differently because their statistical structure differs.

Isolation Forest

Used where multivariate numeric behavior makes sense, especially for rich transaction logs. It is appropriate when anomalies are unusual combinations of otherwise normal-looking fields.

  • Primary use: conn.log, http.log, files.log, some TLS-style logs
  • Score meaning: more isolated rows get higher anomaly score

Rarity scoring

Used for event-like or inventory-like logs where the strongest signal is not geometric distance but uncommon values, combinations, or states.

  • Primary use: weird, notice, known_hosts, known_services, software, arp, packet_filter
  • Score meaning: rarer rows score higher

Hybrid DNS scoring

DNS uses custom logic rather than a generic unsupervised model. It explicitly boosts DGA-like lexical behavior, repeated suspicious query behavior per source, missing answers, and rejection patterns, while downweighting benign local-discovery traffic.

Hybrid connection scoring

conn.log combines a multivariate baseline with explicit scan-behavior bonuses so that broad port sweeps and failed probe campaigns surface more clearly.

Time-series deviation scoring

Telemetry logs such as stats.log and capture_loss.log use deviation scoring over levels and changes because their rows are not independent flows.

Fallback distance

When the feature space is too small or degenerate for the intended model, the tool falls back to a stable distance-style score rather than failing outright.

Important: every detector yields a numeric score, but these scores are not calibrated across different log types. A conn.log score should not be compared directly to a dns.log score. Within-file percentile ranks are used where cross-file comparison is needed.

Directory Maliciousness Score

The final directory score is not a raw sum of anomaly scores. It is a weighted summary of several normalized behaviors.

core_score = 100 * ( 0.35 * weighted_top + 0.25 * uid_corr_score + 0.20 * weighted_fraction + 0.15 * weird_notice_bonus + 0.05 * fuid_bonus ) directory_score = min(100, core_score + 45 * behavior_score)

What the components mean

  • weighted_top: how strong the top anomalies are inside each file, after weighting by log importance
  • weighted_fraction: how much anomaly mass is spread across each file, again weighted by log importance
  • uid_corr_score: how many anomalous uid values are shared across multiple log types
  • weird_notice_bonus: extra weight when already-exceptional logs contribute anomalous rows
  • fuid_bonus: extra weight when anomalous HTTP rows overlap with anomalous files through fuid
  • behavior_score: explicit source-level scan behavior from conn.log

Why the behavior score exists

Some attack directories, especially simple scans, do not generate much uid overlap into higher-level logs. A separate behavior stage over conn.log captures source fan-out, failed-connection fraction, short-connection fraction, zero-payload fraction, and maximum per-host port sweep. This is what allows the detector to recognize broad scan campaigns as strongly malicious even when the higher-level logs are sparse.

Normal Baseline Training

The baseline logic is designed for the reality that “normal” Zeek directories still vary a lot.

Why not train on raw anomaly scores?

Raw row-level scores vary across log types, detector families, traffic volumes, and capture duration. Training thresholds directly on those values would be unstable and hard to interpret.

What is actually trained

The tool learns thresholds from directory-summary metrics such as:

  • score
  • weighted_top
  • weighted_fraction
  • uid_corr_score
  • weird_notice_bonus
  • fuid_bonus
  • behavior_score
  • conn_scan_score
  • cross-log overlap counts

How thresholds are learned

  • With multiple normal directories, the tool uses robust statistics: median and MAD-based upper bounds
  • With very few normal directories, it falls back to conservative upper margins above observed normals
Interpretation: the baseline comparison is not asking “is this exact row anomalous?” It is asking whether the directory’s aggregate behavior exceeds what was observed in known-normal directories.

Outputs and Interfaces

Default terminal mode

Prints anomaly tables per file and, in directory mode, prints a final directory summary and optional baseline comparison.

--summary-line

Suppresses the normal terminal output and prints one final tab-separated line with input path, score, severity, and baseline verdict if present. In ANSI-capable terminals, the score and verdict fields are colorized.

JSON export

Exports per-file summaries, anomalous identifiers, top rows, directory summary, and optional baseline comparison.

PDF plot export

Exports a summary page, a combined flow-by-flow normalized score page, and one score plot per analyzed log file. When a baseline is present, the summary plot overlays suspect bars, normal medians, and normal thresholds.

Design Decisions and Why They Were Made

  • Per-log detectors instead of one model: because Zeek logs have incompatible semantics
  • Cross-log context before scoring: because many attacks are only obvious when logs are tied together
  • Explicit DNS and scan logic: because generic unsupervised scoring often misses DGAs and simple sweeps
  • Directory-level scoring from normalized components: because raw anomaly scores do not compare cleanly across logs
  • Robust baseline training: because normal traffic varies and naive fixed thresholds are brittle
  • Ignore loaded_scripts.log: because runtime configuration noise was hurting detection quality more than helping
What the tool does not claim: the final directory score is not a calibrated probability of compromise. It is a triage-oriented maliciousness score meant to help rank directories and focus analyst attention.

How to Use

Single file

python3 zeek-anomaly-detector.py -f conn.log

Directory analysis

python3 zeek-anomaly-detector.py -d /path/to/zeek

With normal baseline

python3 zeek-anomaly-detector.py \
  -d /path/to/suspect \
  -N /path/to/normal1 \
  -N /path/to/normal2

One-line final summary

python3 zeek-anomaly-detector.py \
  -d /path/to/suspect \
  -N /path/to/normal1 \
  --summary-line

Export JSON and plots

python3 zeek-anomaly-detector.py \
  -d /path/to/zeek \
  -J summary.json \
  -P scores.pdf
